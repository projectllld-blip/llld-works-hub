# LLLD Works Market ゴールと進行方針

この文書は、LLLD Works Hubを外部投稿型コンテンツプラットフォームへ拡張するための「ブレ防止用の正本」です。
実装前に毎回読み、今回の作業がどの段階に属するかを確認します。

## 最終ゴール

LLLD Works Hubを、社内便利ポータルから次のようなプラットフォームへ育てる。

- クリエイターが便利ツール、資料、テンプレート、HTMLアプリを投稿できる
- 利用者が無料利用、買い切り、サブスク、問い合わせ購入を選べる
- 利用者がクリエイターに開発依頼できる
- クリエイター同士、利用者同士、運営が交流できる
- LLLDが審査、品質管理、公開管理、販売導線を担う

合言葉:

> 現場で生まれた便利ツールを、必要な人へ届ける。

## 変えてはいけない芯

- 現行の社内Hubを壊さない
- よく使うものへすぐ戻れる
- 最終利用履歴8件の思想を維持する
- 小さい現場の人が迷わず使える
- GitHub Pagesで軽く動く
- localStorageへ重要情報を保存しない
- 最初からログイン、決済、自動投稿を入れない

## 段階

### Phase 0: 足場固め

目的:

- 本番GitHub Pagesを壊さず、ローカルで検証できる状態にする
- コンテンツ情報をHTMLベタ書きからJSON仮データへ移す
- 将来Supabaseへ置き換えやすいService層を作る

完了条件:

- `python3 -m http.server 5500` でトップが動く
- `data/contents.json` から一覧を読める
- `data/authors.json`、`data/categories.json` がある
- `contentService.js` 経由でデータ取得できる
- `storagePolicy.js` でlocalStorage方針が明記されている
- 既存トップページが壊れていない

### Phase 1: マーケット風社内ポータル

目的:

- 社内Hubの便利さを維持したまま、外部投稿型マーケットに見える導線を追加する

追加候補:

- `marketplace.html`
- `content-detail.html`
- `submit.html`
- `author.html`
- 価格ラベル
- 投稿者情報
- 対象ユーザー
- 無料、無料β、社内限定、問い合わせ、準備中の表示
- 投稿募集導線

完了条件:

- 現行Hubから既存コンテンツを迷わず開ける
- 新規マーケット導線がある
- 詳細ページが販売・問い合わせに耐える情報構造になっている
- まだログイン、決済、自動投稿は入っていない

### Phase 2: 審査制投稿

目的:

- 外部投稿を受け付けるが、完全自動公開にはしない
- 著作権、個人情報、品質、料金トラブルを防ぐ

運用:

1. 投稿者がフォーム等で申請
2. LLLDが内容確認
3. サムネイル、説明文、タグを整備
4. JSONまたは管理データへ登録
5. Works Marketに掲載

完了条件:

- 投稿受付フォームがある
- 投稿審査チェックリストがある
- 掲載前にLLLDが確認する運用になっている
- 投稿者情報とコンテンツ情報を分けて管理できる

### Phase 3: Supabase検証

目的:

- 本物のアカウント制、DB、投稿申請、権限管理へ進めるか検証する

検証対象:

- Supabase Auth
- contents
- authors
- categories
- submissions
- favorites
- profiles
- 管理者だけが公開できる仕組み

完了条件:

- Supabase検証環境が本番とは別にある
- JSON仮データからDBへ移せる設計になっている
- 本番Hubを壊さずに検証できる
- アカウント情報、審査状況、購入履歴をlocalStorageに保存していない

### Phase 4: 社内・関係者β

目的:

- 外部公開前に、社内・関係者で実運用テストをする
- 決済より先に、掲載、問い合わせ、利用導線を確認する

載せるもの:

- LLLD公式コンテンツ
- 講師作成コンテンツ
- 知人事業者のサンプル投稿
- 無料公開コンテンツ
- 問い合わせ販売コンテンツ

完了条件:

- 関係者が実際に使える
- 投稿から掲載までの運用が回る
- 問い合わせ導線が機能する
- 不具合、需要、説明不足を収集できる

### Phase 5: 販売・サブスク・開発依頼

目的:

- 収益化する
- クリエイターが販売、依頼受付、継続提供できるようにする

候補:

- Stripe決済リンク
- 銀行振込
- 手動請求
- Stripe Connect
- サブスク
- 開発依頼フォーム
- レビュー

完了条件:

- 購入履歴と権限管理がDB管理されている
- 売上管理がlocalStorageに入っていない
- 返金、問い合わせ、権利関係の運用が決まっている

## 推奨順序

今後しばらくはこの順で進める。

1. Phase 0を完成させる
2. Phase 1のページを追加する
3. Phase 2の投稿受付と審査運用を作る
4. Phase 3のSupabase検証を始める
5. Phase 4で社内・関係者βを回す
6. Phase 5で販売機能を検討する

## 判断に迷ったとき

迷ったら、次の順に優先する。

1. 現行Hubを壊さない
2. 小さい現場の人が迷わない
3. GitHub Pagesで軽く動く
4. データをJSON/Service層に寄せる
5. 将来DBへ移しやすくする
6. 販売・ログイン・自動投稿は後回しにする

## 次にやる具体案

最初に着手する実装は、Phase 0の仕上げ。

- `data/contents.json` をマーケット型項目へ整理
- `data/authors.json` を追加
- `data/categories.json` を追加
- `assets/js/contentService.js` を追加
- `assets/js/authMockService.js` を追加
- `assets/js/storagePolicy.js` を追加
- 現行トップの表示を壊さない

その次に、Phase 1として以下を追加する。

- `content-detail.html`
- `submit.html`
- `author.html`
- 必要なら `marketplace.html`
