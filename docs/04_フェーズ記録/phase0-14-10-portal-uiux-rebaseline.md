# v0.14.10 ポータルUI/UX再ベースライン

## 目的

v0.15 エラー処理・空状態へ進む前に、直近で大きく拡張された `portal.html` のUI/UX、保存状態、ロードマップ上の位置づけを再確認する。

現在のポータルは、単なる社内アプリ入口ではなく、メモ、掲示板、ファイル保管庫、編集モード、Undo / Redo、詳細ポップアップを含む「編集可能な社内作業ポータル」に近づいている。

## 確認したUI/UX変更

- `portal_updated_v18.html` ベースの新 `portal.html` を社内入口として採用
- 小テスト作成、入試問題、PDF編集、SeatFlow、勤怠、面談支援への実リンクを接続
- スマホ幅の横スクロール、最近使ったもの、開くボタン、購入済み表示を調整
- 保管庫リンクと操作ログからページ遷移確認モーダルを表示
- メモ / ToDoの詳細、削除、追加ポップアップを追加
- 掲示板投稿の詳細、削除を追加
- ファイル保管庫に編集モード、削除、名前変更、リンク変更、ファイル差し替えを追加
- PC版でファイル / フォルダをドラッグ移動できるように調整
- ドラッグ中のゴースト表示と移動先フォルダの「ここへ移動」表示を追加
- 保管庫ツリーにUndo / Redoを追加
- メモ / ToDoのダウンロードをアイコン化し、CSV / Excel / PDF / JPG選択式に変更
- 保管庫ツリーの並び順を「フォルダ -> ファイル」に整理

## チャッピー設計とのズレ

ズレあり。

当初の `portal.html` は「社内アプリへの入口」として整理していたが、現在はメモ、掲示板、ファイル保管庫を編集できる作業ポータルに近づいている。一方で、これらの編集操作の多くはDOM上だけの一時状態であり、SupabaseやlocalStorageへの永続保存はまだ行っていない。

特に注意すべきズレは以下。

- 削除、名前変更、移動、リンク変更が「保存された操作」に見えやすい
- メモ / ToDo追加や掲示板削除がリロード後も残るように見えやすい
- Undo / Redoがあることで、保管庫編集が本格保存機能のように見えやすい
- v0.15の既存定義は認証、アカウント、SeatFlow保存エラーが中心で、ポータル編集系の空状態や未保存状態を十分に含んでいない

## 更新後ロードマップ案

### v0.14.10 ポータルUI/UX再ベースライン

現在の `portal.html` の機能、保存状態、mock範囲を整理する。

### v0.14.11 ポータル編集機能の保存方針整理

メモ / ToDo、掲示板、保管庫編集を、当面どこまで一時状態として扱い、どこからlocalStorageやSupabase保存に進めるかを決める。

この段階で最低限決めること。

- リロードで消える操作には画面上の注意を出すか
- localStorageに保存してよいデータと保存してはいけないデータ
- ファイル本体や個人情報を保存しない方針
- 保管庫ツリー構造を将来Supabase保存する場合のテーブル設計候補
- 社内ポータルのmockと実保存をどう見分けるか

### v0.15 エラー処理・空状態

v0.15は以下を含める形に再定義する。

- 認証・アカウント系: login、signup、account、logout、company_accounts、app_instances
- Supabase保存系: app_data保存、app_data読込、SeatFlowクラウド保存、SeatFlowクラウド読込
- ポータル編集系: メモ / ToDo、掲示板、ファイル保管庫、名前変更、リンク変更、ファイル差し替え、ドラッグ移動、Undo / Redo
- mock / 未保存状態: 保存されない操作、リロードで戻る操作、未実装の出力やアップロード

### v0.16 RLS・他社データ混入テスト

RLSテストは、Supabase保存対象が明確になってから進める。現時点では company_accounts、app_instances、app_data のSeatFlow `seat_layout` が主対象。

ポータル編集データをSupabase保存対象に加える場合は、v0.16前にDB設計とRLS方針を追加する。

## v0.15へ進む条件

- ポータル内編集機能が一時状態か永続保存かを明記する
- リロードで消える可能性がある操作には、ユーザーが誤解しない文言を出す
- mock機能と実保存機能の見た目や文言を整理する
- 削除、名前変更、移動、ファイル差し替えの保存範囲を明確にする
- v0.15の対象にポータル編集系のエラー、空状態、未保存状態を含めることを決める

## 判断

先に `v0.14.11 ポータル編集機能の保存方針整理` を挟むことを推奨する。

理由は、現在のポータル編集機能が見た目上はかなり本格的だが、多くは未保存 / mockであり、このままv0.15へ進むと「何のエラー状態を整えるべきか」が曖昧になるため。
