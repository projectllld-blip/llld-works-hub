## v1.3aで実装した範囲
- `account.html` に「アプリ追加申請」エリアを追加。
- `account.html` のheroから「アプリ追加申請を見る」で申請エリアへ移動できる。
- 申請エリアを強調カードにし、利用中アプリ一覧の直後に表示。
- 利用中アプリ一覧の結果をもとに、固定候補カタログから未利用アプリを表示。
- アプリ候補ごとに `利用中` / `未利用` / `申請済み` を表示。
- 未利用アプリの「追加を申請する」ボタンから申請フォームを表示。
- 申請フォームに以下を表示。
  - 申請したいアプリ
  - 利用目的・相談内容
  - 連絡希望
- 送信後、画面内状態だけで申請済みに切り替える。
- mock modeでも申請操作の見た目を確認できる。

## 保存方針
v1.3aでは申請内容をDBへ保存しない。

保存しないもの:
- `app_instances`
- `app_data`
- `company_accounts`
- `plan_status`
- Supabase Auth
- 外部通知
- メール送信
- 決済・契約状態

申請内容はブラウザ上の一時状態だけで保持する。再読み込みすると消える。

## DB保存をしない理由
既存DB構造には、アプリ追加申請を正式に保存するための専用テーブルがない。

`app_instances` は利用中アプリを表すテーブルであり、申請中の相談内容を保存する用途には使わない。

`app_data` はアプリ内データ保存用であり、申請ワークフローの正本として使わない。

## v1.3でやらないこと
- app_instances の自動追加
- apps の本番データ変更
- app_data への申請保存
- company_accounts の契約状態変更
- plan_status 変更
- Supabase migration追加
- RLS変更
- Auth / login / signup変更
- 決済連携
- メール送信
- 管理者承認画面

## 今後の正式保存候補
以下は旧方針の記録。購入者向け導線としては採用しない。

正式な申請ワークフローを扱う場合は、購入者向けサポート申込またはクリエイター向け投稿 / 掲載申請として別Phaseで再設計する。

- `app_add_requests` のような申請専用テーブル
- 申請者の `company_account_id`
- 申請対象 `app_key`
- 相談内容
- 連絡希望
- 申請状態
- 作成日 / 更新日
- RLS方針
- 管理者確認方法

この設計にはDB / RLS / migrationが関わるため、実装前に人間確認が必要。
