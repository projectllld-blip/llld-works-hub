# phase1-3-spec-realignment

## Phase

v1.3 仕様名・ロードマップ見直し

## 背景

v1.3d `account.html` 申請DB保存の人間確認で、以下が確認された。

- `account.html` で申請操作後、再読み込みすると「申請内容はまだ送られていません。」に戻る。
- Supabase Dashboardの `app_add_requests` に新しい行が追加されていない。
- したがって、DB保存は成功していない。
- v1.3dは完了扱いにできない。

さらに、仕様の前提にズレがあった。

これまでのv1.3は「購入者が未利用アプリの利用申請を出す」流れとして進めていたが、本来の「アプリを追加する」は、クリエイター / 開発者がWorks Market上で利用・販売できるアプリを開発・追加する意味。

購入者側は、アプリを購入したら基本的に即時使えるようになる想定。

購入者に必要なのは「利用申請」ではなく、購入後または購入前後の任意サポート申込。

例:

- 初期設定サポート
- 使い方説明
- カスタマイズ相談
- 導入代行

## 決定内容

- v1.3d account.html 申請DB保存は停止する。
- 現在の `app_add_requests` テーブルは削除しない。
- `app_add_requests` はPARKED扱いにする。
- 購入者側の「アプリ追加申請」は不要とする。
- 購入者向けに `app_add_requests` を使う導線は本線から外す。
- 購入者向け導線は「購入 → 利用開始 → 任意サポート申込」として再整理する。
- クリエイター向けの「アプリ追加」は別フェーズとして、アプリ投稿 / 掲載申請機能に分離する。
- Supabase / RLS / migration の追加変更は行わない。
- コード変更が必要な場合は、先に方針docsを人間確認に回す。

## 購入者向け導線

購入者向けの自然な流れ:

```text
アプリを見る
↓
購入する / 利用開始する
↓
利用開始
↓
必要な場合だけサポートを申し込む
```

購入者向けに扱う候補:

- 購入前相談
- 初期設定サポート
- 使い方説明
- カスタマイズ相談
- 導入代行

購入者向けに扱わないもの:

- 既存アプリの利用申請
- アプリ追加申請
- アプリそのものの投稿
- Works Market掲載申請
- 開発者登録
- アプリ審査

## クリエイター向け導線

クリエイター / 開発者向けの自然な流れ:

```text
クリエイター登録 / 相談
↓
アプリ投稿 / 掲載申請
↓
審査
↓
Works Market掲載
↓
販売 / 利用開始
```

これは購入者向け `account.html` 申請導線とは分ける。

候補フェーズ:

- v1.x クリエイター向けアプリ投稿方針整理
- v1.x アプリ投稿 / 掲載申請MVP
- v1.x 運営側 掲載申請確認MVP

## app_add_requestsの扱い

現時点ではdropしない。

理由:

- 既に本番相当DBに作成済み。
- dropは破壊的変更であり、人間判断が必要。
- 将来、購入者向けサポート申込またはクリエイター向け掲載申請へ再設計できる可能性がある。

ただし、現在のまま購入者向け導線の正本テーブルとして進めない。

候補:

- PARKED: 現状維持。使わない。
- 再設計: 購入者向けサポート申込テーブルとして作り直す。
- リネーム/新設: クリエイター向け掲載申請には別テーブルを作る。

推奨:

```text
app_add_requests はPARKED。
購入者向けサポート申込とクリエイター向け掲載申請は、別名・別テーブルで再設計する。
購入者向けの「アプリ追加申請」導線には戻さない。
```

## account.html実装の扱い

mainにはv1.3d実装が入っているが、DB保存は人間確認で失敗している。

対応候補:

### 案A: account.htmlのアプリ追加申請エリアを一時停止表示に戻す

- 最小変更で安全。
- 「現在準備中」と表示し、DB保存を実行しない。
- 購入者向けサポート申込として再設計するまで誤操作を防げる。

### 案B: v1.3aのUI mockへ戻す

- 画面確認用として残せる。
- ただし「アプリ追加申請」という名前が仕様誤解を残す。

### 案C: account.htmlから申請エリアを非表示にする

- 誤操作は最も少ない。
- ただし、後で購入者向けサポート導線を作る必要がある。

推奨:

```text
案A: account.htmlの申請DB保存を止め、一時停止表示にする。
```

ただし、今回はコード変更せず、別PRで対応する。

## 次に整理する本線候補

次の本線候補は、v1.3の名称を改めて以下のどちらかに分ける。

1. 購入者向け: サポート申込 / 導入相談MVP
2. クリエイター向け: アプリ投稿 / 掲載申請 方針整理

購入ページや決済へ進む前に、どちらを先に作るか人間判断が必要。

## STOP条件

- `app_add_requests` をdropする必要がある。
- Supabase migration / RLS変更が必要。
- account.htmlのDB保存停止コード変更が必要。
- 購入者向けサポート申込とクリエイター向け掲載申請の優先順位判断が必要。
- 決済 / 購入履歴へ踏み込みそう。
