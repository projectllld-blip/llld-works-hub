# ポータル状態管理・エラー表示ルール

## 目的

`portal.html` が編集可能な社内作業ポータルへ拡張されたため、実保存、mock、一時状態を混同しないための実装ルールを定める。

## 基本方針

- 社員が「保存された」と誤解するUIにしない
- リロードで消える操作は、画面上またはトーストで一時状態だと分かるようにする
- ポータル編集データは最終的にSupabase保存対象にする
- MVPでは `app_data.data_type = portal_state` にポータル用JSONとして保存する
- 保存単位は企業アカウント単位とし、`company_account_id` で分離する
- 個人情報、顧客情報、教材PDF本体、契約書、見積書、秘密情報はポータル内mockやlocalStorageに保存しない
- GitHub Pages単体で完結する範囲と、将来Supabaseへ接続する範囲を分ける

## 状態の分類

### 実リンク

小テスト作成、入試問題、PDF編集、SeatFlow、勤怠、面談支援、login、account、marketplaceなど、実HTMLへ遷移するもの。

ルール:

- 相対パスで指定する
- ページ遷移確認モーダル後に別タブで開く方針を維持する
- 存在しないHTMLへリンクしない

### Supabase保存予定の画面状態

v0.14.12以降、以下は `works_portal` の `app_data.portal_state` へ保存する方針にする。

- メモ / ToDo
- 掲示板投稿
- 保管庫ツリー構造
- 保管庫リンク情報
- お気に入り
- 最近使ったもの
- ポータル表示設定

ルール:

- 保存中 / 保存済み / 保存失敗を画面に出す
- 未ログイン時は「ログインすると会社ごとのポータル状態を保存できます」と案内する
- Supabase未設定時はmock表示として扱う
- ファイル本体は保存せず、リンクURLとメタデータだけを保存する

### 一時状態

Undo / Redo履歴、ドラッグ中状態、モーダル開閉状態、一時選択状態、ダウンロード形式選択など、保存しないもの。

ルール:

- 「この画面内だけの変更です」または同等の注意を出す
- リロード後に戻る可能性があることを隠さない
- 実保存と同じ文言を使わない
- `保存しました` ではなく `画面上で反映しました` のような文言にする

### mock

ファイルアップロード本保存、フォルダ作成本保存、購入済みコンテンツDB接続、ToDoダウンロード本出力など。

ルール:

- `data-mock="true"` または説明文でmockであることを分かるようにする
- クリックできるのに何も起きない状態を避ける
- 後続フェーズ予定のトーストを表示する

### Supabase保存

v0.14.12以降の保存対象は以下。

- `company_accounts`
- `app_instances`
- SeatFlowの `app_data.data_type = seat_layout`
- Works Portalの `app_data.data_type = portal_state`

ルール:

- service role keyをフロントに置かない
- RLS未確認のテーブルを本番運用しない
- 企業アカウント単位の `company_account_id` 分離を守る
- ポータル編集データは `app_key = works_portal` / `data_type = portal_state` で扱う
- 専用テーブル化はPhase 2以降の拡張候補とし、MVPではJSONB保存を優先する

## v0.15で扱うエラー・空状態

### 認証・アカウント系

- 未ログイン
- ログイン失敗
- メール未確認
- company_accountsなし
- app_instancesなし
- logout失敗

### Supabase保存系

- Supabase未設定
- app_data保存失敗
- app_data読込失敗
- SeatFlow app_instanceなし
- RLS拒否の可能性があるエラー

### ポータル編集系

- メモ / ToDoが空
- 掲示板投稿が空
- portal_state読込失敗
- portal_state保存失敗
- works_portal app_instanceなし
- 削除対象が未選択
- 保管庫の移動先が不正
- Undo / Redoできる履歴がない
- リンクURLが未入力
- ファイル差し替え対象が未選択
- mock操作であることの通知

## v0.15で追加した表示ルール

- Portalはヘッダーの保存バッジとは別に、次の操作が分かる通知バーを出す
- 読込失敗時は「再試行」導線を出す
- 保存失敗時は「保存を再試行」導線を出す
- localStorageキャッシュのみ表示中は「前回保存データを一時表示中」と明記する
- ページ読込直後のキャッシュ表示だけでSupabaseへ保存しない
- 未保存変更がある場合は保存済みに見せない
- ファイル追加欄では「ファイル本体は保存しない。ファイル名・種別・リンクURL・表示情報のみ保存」と明記する
- account画面には再読込導線を置く
- SeatFlowは未ログイン / app_instanceなし / 未保存 / 保存失敗 / 読込失敗ごとに確認先を表示する

## 禁止事項

- 一時状態の操作に対して、永続保存済みのような表現を使わない
- ファイル本体をlocalStorageへ保存しない
- 個人情報や顧客情報を検証用データに入れない
- Supabase DB構造を決めずにポータル編集データを保存し始めない
- RLS設計なしにポータル編集テーブルを追加しない

## v0.14.11で決めること

- メモ / ToDo、掲示板、保管庫ツリー、お気に入り、最近使ったものはSupabase `app_data.portal_state` に保存する
- ファイル本体はv1.0までは急がず、リンクURL・ファイル名・メタデータ保存を優先する
- Supabase Storageは別フェーズで設計する
- v0.14.12で `portalStateService.js` を追加し、読込 / 保存MVPに進む
