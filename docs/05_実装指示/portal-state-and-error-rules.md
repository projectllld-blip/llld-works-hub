# ポータル状態管理・エラー表示ルール

## 目的

`portal.html` が編集可能な社内作業ポータルへ拡張されたため、実保存、mock、一時状態を混同しないための実装ルールを定める。

## 基本方針

- 社員が「保存された」と誤解するUIにしない
- リロードで消える操作は、画面上またはトーストで一時状態だと分かるようにする
- Supabase保存対象は、設計とRLSが整ったものだけに限定する
- 個人情報、顧客情報、教材PDF本体、契約書、見積書、秘密情報はポータル内mockやlocalStorageに保存しない
- GitHub Pages単体で完結する範囲と、将来Supabaseへ接続する範囲を分ける

## 状態の分類

### 実リンク

小テスト作成、入試問題、PDF編集、SeatFlow、勤怠、面談支援、login、account、marketplaceなど、実HTMLへ遷移するもの。

ルール:

- 相対パスで指定する
- ページ遷移確認モーダル後に別タブで開く方針を維持する
- 存在しないHTMLへリンクしない

### 一時状態

メモ / ToDo追加、掲示板削除、保管庫の名前変更、リンク変更、ドラッグ移動、Undo / Redoなど、現在DOM上だけで動くもの。

ルール:

- 「この画面内だけの変更です」または同等の注意を出す
- リロード後に戻る可能性があることを隠さない
- 実保存と同じ文言を使わない
- `保存しました` ではなく `画面上で反映しました` のような文言にする

### mock

ファイルアップロード本保存、フォルダ作成本保存、購入済みコンテンツDB接続、ToDoダウンロード本出力など。

ルール:

- `data-mock="true"` または説明文でmockであることを分かるようにする
- クリックできるのに何も起きない状態を避ける
- 後続フェーズ予定のトーストを表示する

### Supabase保存

現時点では以下に限定する。

- `company_accounts`
- `app_instances`
- SeatFlowの `app_data.data_type = seat_layout`

ルール:

- service role keyをフロントに置かない
- RLS未確認のテーブルを本番運用しない
- 企業アカウント単位の `company_account_id` 分離を守る
- ポータル編集データをSupabaseへ入れる場合は、先にDB設計とRLS方針を追加する

## v0.15で扱うエラー・空状態

### 認証・アカウント系

- 未ログイン
- ログイン失敗
- メール未確認
- company_accountsなし
- app_instancesなし
- logout失敗

### Supabase保存系

- Supabase未設定
- app_data保存失敗
- app_data読込失敗
- SeatFlow app_instanceなし
- RLS拒否の可能性があるエラー

### ポータル編集系

- メモ / ToDoが空
- 掲示板投稿が空
- 削除対象が未選択
- 保管庫の移動先が不正
- Undo / Redoできる履歴がない
- リンクURLが未入力
- ファイル差し替え対象が未選択
- mock操作であることの通知

## 禁止事項

- 一時状態の操作に対して、永続保存済みのような表現を使わない
- ファイル本体をlocalStorageへ保存しない
- 個人情報や顧客情報を検証用データに入れない
- Supabase DB構造を決めずにポータル編集データを保存し始めない
- RLS設計なしにポータル編集テーブルを追加しない
