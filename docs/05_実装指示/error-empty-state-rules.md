# エラー処理・空状態ルール

## 基本方針

画面上のエラーは、現場の人が次に何をすればよいか分かる言葉にする。

開発者向けの細かい原因は、docs、検証ログ、Supabase Dashboard確認手順に残す。

## 表示する状態

- 読み込み中
- 未ログイン
- Supabase未設定
- Supabase client未準備
- 企業アカウント未作成
- 利用アプリなし
- クラウド保存データなし
- 保存失敗
- 読込失敗

## ユーザー向け文言

- 専門用語を出しすぎない。
- ただし、確認が必要な場所は分かるようにする。
- 例: `登録時に選んだアプリ、またはSupabaseのapp_instances設定を確認してください。`

## 開発者向け情報

画面にはキー、token、SQL詳細、RLS policy本文を出さない。

必要な情報は以下に分ける。

- Supabase Dashboardで確認すること
- docsに残すこと
- Codex実行ログで確認すること

## mock mode

- mock modeではDB保存を実行しない。
- mock modeであることを表示する。
- 入力内容、パスワード、認証トークンをlocalStorageへ保存しない。

## supabase mode

- Supabase未設定時はmock modeへ安全に戻す。
- Supabase設定済みでも、未ログインなら保存・読込しない。
- `company_accounts` がなければ、自動作成せず、signup triggerやRLSを確認する。
- `app_instances` がなければ、自動作成せず、登録時の選択アプリやv0.13 migrationを確認する。
- `app_data` がなければ、初回保存を案内する。

## SeatFlow

- クラウド保存中、読込中はボタンを一時的に無効化する。
- 初回読込でデータがない場合は、先にクラウド保存するよう案内する。
- 保存失敗、読込失敗は同期ラベルにも表示する。
- 生徒名、講師名、利用状況、勤怠情報はクラウド保存しない。
