# LLLD Works Market 毎回確認チェックリスト

LLLD Works Hub / Marketを変更するたびに、このチェックリストで確認する。  
目的は「その場の勢いで大きく作りすぎない」「本番Hubを壊さない」「将来の販売プラットフォーム化からブレない」こと。

## 1. 作業前チェック

- 今回の作業はどのPhaseか
- 本番GitHub Pagesの社内利用を壊さないか
- UIを大きく変える必要が本当にあるか
- 既存の「よく使うもの」「最終利用履歴8件」を壊さないか
- localStorageへ重要情報を保存しようとしていないか
- HTMLにコンテンツ情報をベタ書きしていないか
- 既存の未コミット変更を混ぜない準備ができているか

## 2. データ設計チェック

- コンテンツ情報は `data/contents.json` にあるか
- 投稿者情報は `data/authors.json` にあるか
- カテゴリ情報は `data/categories.json` にあるか
- 画面側は `contentService.js` 経由で取得できるか
- 将来Supabaseへ置き換える前提の関数名になっているか
- コンテンツID、slug、authorId、categoryIdが対応しているか
- 公開状態、価格種別、対象ユーザー、タグが分離されているか

## 3. localStorageチェック

localStorageに保存してよいもの:

- 最近見たコンテンツ
- 最終利用履歴8件
- 表示設定
- 一時的な下書き
- ログイン不要ツールの一時作業データ

localStorageに保存しないもの:

- アカウント情報
- 投稿者情報
- 購入履歴
- 審査状況
- 公開/非公開ステータス
- 権限情報
- 売上情報
- 個人情報を含む業務データ

## 4. UI/UXチェック

- トップページから既存アプリを迷わず開けるか
- カードが見やすいか
- 価格やステータス表示が誤解を生まないか
- 投稿者情報が必要以上に前に出すぎていないか
- 社内ポータルとしての使いやすさが残っているか
- 外部向けに見せても意味が通る言葉になっているか
- 日本の小規模事業者、塾、店舗向けの言葉になっているか
- 説明文が増えすぎていないか

## 5. リンク・パスチェック

- `python3 -m http.server 5500` で起動できるか
- `http://localhost:5500/` でトップが表示されるか
- CSS、JS、JSONが200で読めるか
- 画像、サムネイル、内部リンクに404がないか
- GitHub Pages専用の絶対パスが混ざっていないか
- 相対パス中心になっているか

## 6. フェーズ別チェック

### Phase 0

- JSON仮データが読める
- Service層がある
- localStorage方針が明記されている
- 既存Hubの表示が壊れていない

### Phase 1

- 詳細ページがある
- 投稿募集ページがある
- 投稿者ページがある
- 価格ラベルがある
- まだログイン・決済・自動投稿を入れていない

### Phase 2

- 投稿受付フォームがある
- 審査前提の運用になっている
- 著作権・個人情報チェックがある
- LLLDが公開判断できる

### Phase 3

- Supabase検証環境が本番と分かれている
- アカウント、投稿者、投稿申請、公開状態がDB想定になっている
- localStorageに権限情報を置いていない

### Phase 4

- 現在従業員がいない場合は、代表本人による疑似βチェックに圧縮する
- 問い合わせ導線がある
- 決済なしでも運用確認できる
- フィードバックを集める場所がある

### Phase 5

- Supabase移行準備だけを行う
- 本番Supabase接続はまだしない
- APIキー、秘密キー、サービスロールキーをHTML/JS/JSONに置いていない
- `ContentService` 経由で差し替える方針が残っている
- `data/` は仮DBとして維持している

### Phase 6

- 管理画面はSkeletonまたはモックに留める
- 保存、公開、削除、権限変更、決済処理を実装していない
- localStorageに本番権限や審査状況を保存していない
- 投稿管理、審査、販売管理の将来構造だけ確認できる
- 詳細、審査、編集モック、公開判定は画面上の確認だけで、実データを書き換えない
- 投稿申請一覧はモックデータで、個人情報や実メールアドレスを置かない

### Phase 6.2

- 商品詳細で「誰向けか」「解決する悩み」「主な機能」「使い方」「注意事項」が分かる
- 価格、販売状態、提供方法、CTAが矛盾していない
- requestページで相談対象が分かる
- 有料商品の本体ファイルをGitHub Pagesに置いていない
- 決済、認証、DB接続、自動納品を実装していない

## 7. 完了前チェック

- `git diff --check` が通る
- JSON構文チェックが通る
- JS構文チェックが通る
- localhostで表示確認した
- `admin.html` が本番操作不可のモックとして表示される
- 追加したリンクが大きく壊れていない
- 変更ファイルが今回の目的に絞られている
- 不要な大規模リファクタリングをしていない
- GitHubへcommit/pushした

## 8. 完了報告テンプレート

完了報告では次を必ず書く。

- 今回のPhase
- 変更点
- 変更したファイル
- 確認した動作
- 未対応事項
- 残リスク
- GitHub反映状況
